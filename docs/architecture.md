# System Architecture and Protocol Contracts

This document defines the architectural boundaries, communication protocols, and standardized contracts for the mcp-experto-filesystem server.

## 1. Protocol and Distribution

### 1.1 Local Execution via stdio
Given its nature as a filesystem automation tool, the MCP server must always operate via stdio. 
This ensures:
* **Security:** No network ports are opened on the host machine.
* **Locality:** The agent executing the commands runs in the same environment or container as the codebase.
* **Simplicity:** No complex authentication or network routing is required.

## 2. Initialization and Self-Discovery

To prevent LLMs from guessing how to use the server, the system provides aggressive self-discovery mechanisms.

### 2.1 The initialize Hook
Upon connection (the MCP initialize request), the server must proactively return a comprehensive Getting Started payload. This payload will contain a brief summary of all available tools, their purposes, and basic rules of engagement.

### 2.2 The get_help Tool
A dedicated get_help tool must be exposed to allow agents to query documentation dynamically without leaving their context.

* **get_help()** (no arguments): Returns the full initialization documentation (overview of all tools).
* **get_help(topic="standards")**: Returns the strict instructions on how to parse and understand the standard response payload.
* **get_help(topic="[Tool Name]")**: Returns detailed, specific instructions on how to use a particular tool, including arguments, expected output, and edge cases.

## 3. Standard Response Payload (The Universal Contract)

Every tool exposed by this MCP server must return a strictly typed JSON response following the Universal Contract. This ensures the AI agent can parse responses predictably, regardless of the tool used.

### 3.1 Payload Schema

```json
{
  "status": 200,
  "message": "Operation completed successfully.",
  "data": {},
  "error": null,
  "meta": {
    "warnings": [],
    "next_steps": []
  },
  "metrics": {
    "execution_time_ms": 120.5,
    "approx_input_tokens": 45,
    "approx_output_tokens": 300,
    "input_bytes": 150,
    "output_bytes": 1200
  }
}
```

### 3.2 Field Definitions

* **status (integer):** HTTP-like status codes indicating the result of the operation.
    * 200: Success.
    * 400: Bad Request (invalid tool arguments).
    * 403: Forbidden (safety rule violation).
    * 404: Not Found (file or directory does not exist).
    * 500: Internal Server Error.
* **message (string):** A brief, human-readable summary of the operation's result.
* **data (object | array):** The actual payload returned by the tool. Must be empty if an error occurs.
* **error (object | null):** Present only if status >= 400. Contains specific details about the failure.
* **meta (object):** Contains warnings and next_steps to guide the agent.
* **metrics (object):** Telemetry data to monitor token economy and performance.
    * execution_time_ms (float): Tool execution time in milliseconds.
    * approx_input_tokens (integer): Estimated tokens consumed by the input payload.
    * approx_output_tokens (integer): Estimated tokens generated by the output payload.
    * input_bytes (integer): Size of the request arguments.
    * output_bytes (integer): Size of the response.

## 4. Design Philosophy

By forcing all tools through this Universal Contract, we ensure that:
1. The LLM never has to parse unstructured text dumps.
2. Token consumption is transparent and observable.
3. Errors are handled gracefully with clear status codes rather than stack traces.
